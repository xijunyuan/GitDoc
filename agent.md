# Agent.md

## 项目名称
**GitDoc**（暂定）—— 基于 Git 的文档协作系统

## 项目愿景
为课题组、工程项目部等需要**异步协作、强版本管理**的团队，提供一套兼容 Word/WPS，且具备 Git 式分支、PR、合并与历史追溯能力的文档协作工具。用户无需学习 Git，即可享受代码级版本控制带来的清晰与安全。

## 核心原则
- **最小增量**：不取代 Word/WPS，而是用 Git 的协作理念“武装”现有工作流。
- **用户零学习成本**：分支、PR 等概念全部翻译为“评审分支”“请求合并到主文档”等文档术语。
- **异步优先**：刻意不做实时协同编辑，聚焦异步场景下的版本混乱和合并痛苦。
- **兼容 Word/WPS**：以 `.docx` 为主要支持格式，通过解析底层 OOXML 实现结构化对比与合并。

## 目标用户与场景
1. **课题组写论文**：导师、学生多人并行撰写，需逐句审阅、有据可查、可回溯历史。
2. **工程项目部写报告**：多分包/多角色共同撰文，需责任到人、强审批流、稳定主分支。

## 产品形态
最终形态：**本地插件（WPS/Word）+ 远程协作服务器 + Web 端差异审阅界面**。
MVP 形态：**本地版 Word/WPS 版本控制插件**（单机，不依赖服务器）。

## 技术路线总览
### 核心挑战
Git 为纯文本设计，而 `.docx` 是二进制格式（ZIP 包内的 XML）。必须解决：
- 如何将文档内容**文本化、可 diff、可自动合并**。
- 如何将 Git 冲突转化为用户看得懂的**可视化合并界面**。

### 技术决策
采用**解析 OOXML 路线**：不改变用户编辑习惯，在保存时自动解压 `.docx`，提取核心 XML 内容（如 `document.xml`）进行差异计算和版本追踪。样式、批注、表格等复杂元素也需纳入对比范围，提升对比精度。

### 架构概要
- **客户端（插件）**：
  - 监听到保存动作，自动解析文档内容，生成 Git commit。
  - 提供“创建评审分支”“请求合并”“查看历史”等按钮，底层调用 Git 命令。
- **协作服务器**（MVP 后引入）：
  - 轻量 Git 托管，负责 PR 管理、差异对比服务、权限控制。
  - 不提供在线编辑，只做“文档版 GitHub”。
- **差异对比与合并引擎**：
  - 使用 `python-docx` 等库解析 XML 为段落/文本序列。
  - 实现段落级、句子级、甚至词级的精确 diff，并将结果格式化为类似 Word “修订模式” 的输出。
- **冲突解决 UI**：
  - 将 Git 冲突转化为“A修改 vs B修改”的选择界面，用户手动确认或编辑后，系统完成合并。

## MVP 范围（第一阶段）
**目标**：最快验证核心价值——让用户能在 Word/WPS 内无感使用版本管理。
**功能**：
1. **自动版本记录**：监听保存事件，自动生成 commit，并可附带简要说明。
2. **可视化版本历史**：在插件面板中展示时间轴，支持预览旧版本内容。
3. **任意版本差异对比**：选择两个版本，高亮显示文字增删改，精确到字。
4. **一键回滚**：将当前文档恢复到任一历史版本。

**不包含**：远程协作、分支、PR、多人同步。

## 后续迭代路径
- **第二阶段**：本地多人协作。
  - 引入分支（评审分支）、本地 PR（基于本地仓库的合并请求）。
  - 实现冲突解决 UI，支持合并时解决内容冲突。
- **第三阶段**：远程协作。
  - 开发协作服务器，支持推送/拉取分支、远程 PR、Web 端审阅与合并。
  - 针对论文场景实现“导师评审模式”；针对工程场景实现“多源报告组装”（子模块思维）。

## 关键竞品参考
- **Simul Docs**：证明了解析 OOXML 路线的商业可行性，重点关注其冲突合并体验。
- **HedgeDoc**：说明纯文本路径的轻量与开放，可参考其版本历史展示方式。
- **Litera Compare**：法律文档比对标杆，对比引擎的准确度是壁垒。
- **Notion / Google Docs**：启发如何将版本管理包装得极简。

## 技术栈建议
- **文档解析**：Python（`python-docx`, `lxml`）提取 OOXML 内容并进行 diff。
- **Diff 算法**：`difflib` 或 `diff-match-patch` 实现词级对比。
- **插件前端**：WPS 加载项可用 HTML/JS 技术栈（基于 WPS 开放平台），Word 插件可用 Office.js 或 VSTO。
- **Git 操作**：在客户端调用本地 Git 命令行（`git`），或使用 `isomorphic-git` 等纯 JS 实现（便于跨平台）。
- **协作服务器**（后期）：Node.js 或 Go，提供 REST API，内嵌 Git 操作。
- **Web 前端**：React 或 Vue，构建 PR 审阅与差异对比界面。

## 开发指令（给 Claude Code 的上下文）
以下指令供 AI 开发助手参考：

*项目定位*：开发一个文档版本控制插件，目标是让 Word/WPS 用户像程序员使用 Git 一样管理文档历史、分支和合并，但完全隐藏 Git 细节。

*MVP 任务清单*：
1. 搭建 WPS 插件项目基础结构（如使用 WPS 加载项开发工具）。
2. 实现“保存时自动创建 Git commit”逻辑：在后台初始化 Git 仓库，监听文档保存事件，将 `.docx` 解压并提取文本内容（建议同时保留完整文件作为 binary backup），生成 commit。
3. 插件面板 UI：显示版本历史列表（作者、时间、消息），点击后可查看详情。
4. 差异对比功能：选取两个版本，调用 diff 算法，生成带高亮的差异视图，展示在插件面板或新窗口中。
5. 回滚功能：用户确认后，将当前文档内容替换为指定版本。

*注意事项*：
- 务必保证文件编码和格式无损，尤其是在回滚操作中。
- 文本提取需包含段落、列表、表格文本，批注和格式暂可忽略（MVP 后完善）。
- 设计上保持“无感”，插件应轻量且不打断写作流。

*后续扩展预留*：
- Git 分支相关操作（创建、切换、合并）的 UI 接口。
- 与远程仓库交互的接口（push/pull/fetch）。
- PR 模型设计（源分支、目标分支、审阅状态）。
